Waarom over spaghetti code?

Een paar weken geleden kwam ik in contact met een recruiter die zich in het dagelijks werkleven onder andere bezig houdt met het werven van software engineers voor een klein software ontwikkelings bedrijf. Het gesprek nam op een gegeven moment een wending in de richting van de kansen op de arbeidsmarkt voor startende developers met een omscholingstraject als basis. De betreffende persoon (n=1 in mijn analyse) gaf aan kritisch te zijn op startende omscholers en als antwoord op mijn vraag waarom dit het geval was kreeg ik onder andere het schrijven van spaghetti code als antwoord. Alleen al dit metafoor interigeerde me en ik wilde hier wel wat meer van van weten. Deze eerste opdracht voor het maken van website staat daarom in het teken van mijn kleinschalige web research naar het fenomeen spaghetti code. Wat is het, waarom is het onwenselijk, hoe los je het op?

Wat is spaghetti code?

Spaghetti code wordt als metafoor gebruikt voor ongestructueerdere en moeilijk te onderhouden source code. Net zoals een bord spaghetti ziet spaghetti code eruit als een verwarde brei, ingewikkeld om te volgen, en zeker voor anderen moeilijk te begrijpen.

Oorzaken zijn:

  • Te weinig ervaring met Object Oriented Programming/ Design.
  • Ineffectieve code reviews.
  • Het OOP design te weinig perioriteit geven binnen het project.
  • Werkomgeving waarbinnen programmeurs regelmatig afgezonderd werken aan hun deel van het project.
Bron: Sourcemaking

Hierdoor is code op een gegeven moment niet meer leesbaar of zelfs niet meer werkbaar. Wanneer dit laatste het geval is, is spaghetti code niet alleen kostbaar en vervelend voor betrokken programmeurs maar kunnen er disfunctionerende systemen ontstaan met alle gevolgen van dien.

Zembla maakte in 2015 een uitzending over spaghetti code bij het SVB die een systeem gebouwd op spaghetti code aankochten bij Capgemini. Het systeem blijkt niet te werken en de overheid trekt de stekker uit het systeem zonder dat het een dag gefunctioneerd heeft. Schade loopt in de tientallen miljoenen eurso welke met belastingsgeld zijn betaald. Voor meer informatie klik onder het Zembla logo in de afbeelding.

Hoe hou ik mijn code dan schoon?

De impact van spaghetti code is dus potentieel groot en er zijn verschillende veroorzakers. Onervarenheid is hier een van. Van de recruiter kreeg ik mee: Oefenen, oefenen, oefenen. En leren in de praktijk, dit is iets wat alle developers, elke dag weer binnen hun organisatie doen.

Daarnaast vond ik op bron: medium.com de 5S principles, een onderdeel van het Japanse Total Productive Maintenance (TPM) een kwaliteitsbenadering waarbinnen de focus ligt op onderhoud in plaats van op productie:

5S PRINCIPLES

  • Seiri (organization) Gemakkelijk terug te vinden, het geven van logische en unieke namen helpt hierbij.
  • Seito (systematize) De betreffende code moet erug te vinden zijn op een plek waar je deze ook kan verwachten.
  • Seiso (cleaning) Codeer en benoem alleen wat strikt noodzakelijk is. Onnodige informatie veroorzaakt mentale onrust.
  • Seiketsu (standardization) Er is consensus binnen de groep over hoe code gestrutureerd is en iedereen houdt zich hieraan.
  • Shutsuke (discipline) Je houdt je aan de, in de groep gemaakte afspraken, reflecteert regelmatig en wil, indien noodzakelijk, aanpassen.

Erg interessant allemaal, in beginsel toch maar focussen op de tip van de recruiter: De basis oefenen, oefenen, oefenen en voor nu mijn spaghetti code omarmen. Alvast informatie verzamelen kan natuurlijk geen kwaad. Mocht je goede tips hebben praktisch of voor een leeslijst, laat deze dan vooral achter via het tips formulier.
